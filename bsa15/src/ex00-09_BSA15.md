## Exercise 02 — Актуализация UC

_Проанализируй и при необходимости доработай UC04 «Зарегистрировать клиента (менеджер)»_

Контекст использования: Менеджер, при общении с клиентом по телефону, регистрирует клиента, чтобы записать на обслуживание.

Область действия: барбершоп, оказание услуг клиентам.

Уровень: цель пользователя.

Основное действующее лицо: менеджер.

Предусловия: Клиент позвонил менеджеру. Менеджер зашел на сайт барбершопа. Актуальный справочник каналов связи загружен Клиентом с Сервера для отображения в интерфейсе.

### Основной сценарий:

1. Менеджер открывает страницу регистрации клиента
	Приложение: клиент (фронтенд)
	
	Роль: менеджер
	
2. Менеджер вводит номер телефона, имя (и опционально фамилию) клиента, выбирает предпочитаемый канал связи. Нажимает кнопку "Зарегистрировать"
	Приложение: клиент (фронтенд)
	
	Роль: менеджер
	
	Примечание по интеграции: на Клиенте это поля first_name (обязательно) и last_name (опционально), а на Сервере ожидается единое поле full_name
	
3. Система подтверждает, что телефон не зарегистрирован в системе, и регистрирует клиента.
	Приложение: клиент -> сервер (бэкенд)
	
	Роль: система
	
	Интеграционный метод: POST /api/v1/clients 
	
	Клиент отправляет POST-запрос. Сервер проверяет номер телефона в БД. Если телефон уникален, Сервер создает запись в Persona (склеивая имя и фамилию в full_name, устанавливая роль 'client', генерируя логин и temporary_password) и запись в Clients (с выбранным каналом связи). 
	Также в теле запроса передается строка из справочника, например: "channel": "telegram")
	
4. Система генерирует временный пароль, направляет СМС по телефону клиента о регистрации с указанием имени, логина, временного пароля.
	Приложение: сервер (бэкенд) -> внешний сервис (SMS-шлюз)
	
	Роль: система
	
	Действия: POST /api/v1/external/sms/send (вызов внешнего сервиса рассылки).
	
	Описание интеграции: сервер формирует текст сообщения и передает его внешнему шлюзу.
	Интеграционный метод (внешний): API шлюза отправки SMS.
	
5. Система сообщает менеджеру об успешной регистрации клиента
	
	Приложение: сервер -> клиент (фронтенд)
	
	 Роль: система
	 
	 Интеграционный метод: ответ 201 Created на запрос POST /api/v1/clients.
	 
	 Описание интеграции: сервер возвращает Клиенту данные созданного клиента (ID, логин). Клиент отображает на экране сообщение: "Клиент успешно зарегистрирован. Данные для входа отправлены в СМС".

#### Альтернативный сценарий:

2а. Введенный телефон зарегистрирован в системе:
2а1. Система сообщает, что такой номер телефона зарегистрирован в системе
	Приложение: сервер -> клиент (фронтенд)
	Роль: система
	Интеграционный метод: ответ 409 Conflict на запрос POST /api/v1/clients с указанием ошибки "Телефон уже зарегистрирован"
	Приложение: клиент (фронтенд) отображает ошибку менеджеру.
	
2а2. Менеджер вводит другой номер телефона (возврат к шагу 2) или клиент отказывается от регистрации
	Приложение: клиент (фронтенд)
	Роль: менеджер

Справочники, используемые в UC:

Справочник каналов связи: сервер: ENUM('sms', 'telegram', 'whatsapp', 'vk') в поле preferred_notification_channel; Клиент: VARCHAR(15) с комментарием "Предпочитаемый канал: Telegram/WhatsApp/VK/SMS"

Решение по интеграции справочников:
Актуальный справочник передается до выполнения UC.

Если мы выберем передачу значения вместе с исходным сообщением (Вариант 2), фронтенд должен будет "зашить" (хардкод) список каналов в свой код. Если владелец барбершопа решит добавить канал "Viber", придется переписывать и переукладывать фронтенд.  
Если фронтенд при открытии страницы запрашивает справочник с бэкенда (Вариант 1), он всегда будет отображать актуальный список, и при добавлении "Viber" в ENUM на сервере, на фронтенде он просто появится в выпадающем списке.

## Exercise 03 — Интеграция справочников

#### Используемые справочники: 

1. Справочник каналов связи / уведомлений (Clients.preferred_notification_channel / клиент.канал связи)
	Сервер: ENUM('sms', 'telegram', 'whatsapp', 'vk')
	Клиент: VARCHAR(15) (допустимые значения: Telegram/WhatsApp/VK/SMS)

#### Потребность в интеграции

Справочник каналов связи: интеграция **нужна**
Менеджер на фронтенде (Клиенте) должен выбрать канал связи из выпадающего списка, чтобы передать его на сервер. Чтобы фронтенд отображал актуальный список доступных каналов (совпадающий с ENUM-ом сервера), ему нужно получить эти данные от сервера. Разница в типах данных (ENUM vs VARCHAR) также требует согласования через API.

#### Вариант интеграции для справочника каналов связи:

Передача полного справочника

Справочник каналов связи очень мал (4 значения) и меняется редко (только если барбершоп решит интегрировать новый мессенджер). 
Передача дельты или поиск записи нецелесообразны для такого объема. Фронтенду проще и логичнее запросить весь список целиком при открытии формы регистрации и закэшировать его на время пользовательской сессии.

#### Метод интеграции для справочника каналов связи

Для передачи полного справочника фронтенд (Клиент) запрашивает его у бэкенда (Сервера) при инициализации формы

Метод HTTP: GET 
URL-путь: /api/v1/dictionaries/notification-channels
Параметры: отсутствуют
Описание тела ответа (успешный сценарий):
	Код состояния: 200 OK
	Тело ответа: массив объектов JSON, где каждый объект содержит код канала (для передачи в запросе регистрации) и название (для отображения менеджеру)

## Exercise 04 — Определение и описание методов

3 ключевых интеграционных взаимодействия:

1. Получение справочника каналов связи при открытии формы
2. Регистрация клиента (отправка формы)
3. Отправка SMS с учетными данными (внешний вызов от Сервера к SMS-шлюзу)

Вопрос: как фронтенду получить список доступных каналов связи для отображения в выпадающем списке формы регистрации?
Ответ: фронтенд запрашивает у бэкенда справочник каналов связи при открытии страницы.

Определение HTTP-метода:
CRUD операция: Read (чтение)

Безопасный метод: да, требуется. Метод не должен менять состояние сервера.
Идемпотентный метод: да, требуется. Многократный запрос должен возвращать один и тот же результат.

Итоговый метод: GET

Описание запроса:
Метод: GET
URL-путь: /api/v1/dictionaries/notification-channels
Параметры: отсутствуют
Тело запроса: отсутствует 

Набор ответов:

Успешный ответ:
Код состояния: 200 OK
Тело ответа: JSON-массив объектов, содержащих код канала (для передачи в следующих запросах) и отображаемое имя (для UI).

Неуспешный ответ:
Код состояния: 500 Internal Server Error
Тело ответа: JSON с описанием ошибки (например, справочник недоступен). 
Пример: {"error_code": "DIC_UNAVAILABLE", "message": "Справочник каналов связи временно недоступен"}

#### 2/ Регистрация клиента (отправка формы)

Набор «вопрос — ответы»:
Вопрос: Как фронтенду передать данные нового клиента (телефон, имя, канал связи), чтобы бэкенд проверил уникальность телефона и создал учетную запись?
Ответ: фронтенд отправляет POST-запрос с данными клиента в формате JSON. Бэкенд проверяет телефон, создает записи в Persona и Clients, генерирует пароль и возвращает результат.

Определение HTTP-метода:
CRUD операция: Create (создание)
Безопасный метод: Нет, метод изменяет состояние сервера (создает новые записи в БД).
Идемпотентный метод: Нет. Если отправить один и тот же запрос дважды (с одним номером телефона), система  вернет ошибку дублирования
Итоговый метод: POST

Описание запроса:
Метод: POST
URL-путь: /api/v1/clients
Параметры: отсутствуют.
Тело запроса: JSON-объект, содержащий:
phone (строка, обязательное) - номер телефона клиента
first_name (строка, обязательное) - имя клиента
last_name (строка, необязательное) - фамилия клиента
notification_channel (строка, обязательное) - код канала связи из справочника 

Набор ответов:
Успешный ответ:
Код состояния: 201 Created
Тело ответа: JSON-объект с данными созданного клиента (без пароля). Должен содержать системный ID, логин (который сгенерировал сервер). 
Пример: {"id": 15, "login": "79991234567", "phone": "79991234567", "first_name": "Иван", "notification_channel": "sms"}

Неуспешный ответ (Альтернативный сценарий 2а):
Код состояния: 409 Conflict
Тело ответа: JSON с указанием, что телефон уже существует. 
Пример: {"error_code": "PHONE_ALREADY_EXISTS", "message": "Клиент с таким номером телефона уже зарегистрирован в системе"}

#### 3/ Отправка SMS с учетными данными (внешний вызов от Сервера к SMS-шлюзу)

Определение HTTP-метода:
CRUD операция: Create (создание задачи на отправку сообщения в шлюзе)
Безопасный метод: нет
Идемпотентный метод: нет
Итоговый метод: POST

Описание запроса:
Метод: POST
URL-путь: /api/v1/messages/send (эндпоинт внешнего SMS-шлюза)
Параметры: отсутствуют
Тело запроса: JSON-объект, содержащий: 
recipient_phone (строка, обязательное) - номер клиента
message_text (строка, обязательное) - текст SMS с логином и паролем
 
Набор ответов:

Успешный ответ:
Код состояния: 200 OK (или 202 Accepted, если шлюз ставит сообщение в очередь)

## Exercise 08 — Актуализация UC

_Проанализируй и при необходимости доработай UC02 «Сформировать расписание мастера»_

Контекст использования: менеджер создает расписание работы мастера в виде слотов для выбора клиентами.
Область действия: барбершоп, оказание услуг клиентам.
Уровень: цель пользователя.
Основное действующее лицо: менеджер.
Предусловия: Менеджер получил график работы мастеров. Менеджер зашел на сайт.

#### Основной сценарий:

1. Менеджер открывает страницу ввода слотов мастеров на обслуживание.
	 Приложение: клиент (фронтенд)
	 
	 Роль: менеджер
	 
	 Интеграционный метод: GET /api/v1/employees?role=master 
	 
	 Клиент запрашивает у Сервера список сотрудников, отфильтровав их по роли 'Мастер' из сущности Employee_Roles, чтобы отобразить выпадающий список мастеров на UI). 
	 
	 Опционально: GET /api/v1/slots?date={date} для загрузки уже существующих слотов на выбранную дату.

2. Менеджер выбирает дату и слот (период времени длительностью 1 час, указано начальное время), выбирает мастера из сущности «Сотрудники», которым указана роль «Мастер» в сущности «Роль сотрудника».
	Приложение: клиент (фронтенд)
	
	Роль: менеджер
	
	Комментарий: на Клиенте выбранная дата и время склеиваются в единое поле начало (DATETIME). Сервер впоследствии должен будет разделить его на slot_date (DATE) и start_time (TIME).

3. Система подтверждает, что выбранный слот не забронирован в системе клиентом.
	Приложение: клиент -> сервер (бэкенд)
	
	Роль: система
	
	Интеграционный метод: POST /api/v1/slots 
	
	Клиент отправляет данные для создания слота. Сервер проверяет пересечение с уже существующими слотами со статусом 'Booked' в БД).

4. Менеджер подтверждает закрепление времени за мастером.
	Приложение: клиент (фронтенд)
	
	Роль: менеджер
	
	Комментарий: подтверждение происходит в момент нажатия кнопки "Создать", что инициирует запрос из шага 3. Шаг 4 - это действие пользователя в UI, результатом которого является отправка запроса.

5. Система закрепляет слот: дата, время, мастер.
	Приложение: сервер (бэкенд)
	
	Роль: система
	
	Интеграционный метод: ответ 201 Created на запрос POST /api/v1/slots (Сервер сохраняет слот со статусом 'free, разделяет DATETIME на дату и время, и возвращает данные созданного слота Клиенту).

## Exercise 09 — Определение и описание методов

4 ключевых интеграционных взаимодействия:

1. Получение списка мастеров
2. Создание нового слота
3. Изменение забронированного слота
4. Отправка СМС о переносе визита

#### 1/ Получение списка мастеров

Набор «вопрос — ответы»:
Вопрос: как фронтенду получить список сотрудников, являющихся мастерами, для отображения в выпадающем списке формы?
Ответ: фронтенд запрашивает список сотрудников, передавая параметр фильтрации по роли.

Определение HTTP-метода:
CRUD операция: Read (чтение)

Безопасный метод: Да, требуется

Идемпотентный метод: Да, требуется

Итоговый метод: GET

Описание запроса:
Метод: GET
URL-путь: /api/v1/employees
Параметры: role=master (query-параметр для фильтрации сущности Employee_Roles / role)
Тело запроса: отсутствует

Набор ответов:

Успешный ответ:
Код состояния: 200 OK
Тело ответа: JSON-массив объектов с данными мастеров (id, ФИО/Имя, специализация).
Пример: [{"employee_id": 1, "first_name": "Иван", "specialization": "Мужские стрижки"}]

Неуспешный ответ:
Код состояния: 500 Internal Server Error
Тело ответа: JSON с описанием ошибки. Пример: {"error_code": "INTERNAL_ERROR", "message": "Ошибка получения списка сотрудников"}

#### 2/ Создание нового слота

Набор «вопрос — ответы»:
Вопрос: как фронтенду передать данные о новом слоте (дата, время, мастер), чтобы бэкенд проверил отсутствие брони и создал расписание?
Ответ: фронтенд отправляет POST-запрос с данными слота. бэкенд проверяет пересечение времени и, если всё ок, сохраняет слот.

Определение HTTP-метода:
CRUD операция: Create (создание)
Безопасный метод: нет
Идемпотентный метод: нет
Итоговый метод: POST

Описание запроса:
Метод: POST
URL-путь: /api/v1/slots
Параметры: отсутствуют.
Тело запроса: JSON-объект, содержащий:
staff_id (Integer, обязательное) - ID мастера.
start_at (String/DATETIME, обязательное) - начало слота (Клиент передает единым полем, Сервер сам разбивает на slot_date и start_time).
status (String, опциональное) - по умолчанию Сервер установит "Free".

Набор ответов:
Успешный ответ:
Код состояния: 201 Created
Тело ответа: JSON с созданным слотом
Пример: {"slot_id": 55, "staff_id": 1, "start_at": "2026-06-14T10:00:00Z", "end_at": "2026-06-14T11:00:00Z", "status": "Free"}

Неуспешный ответ (Слот занят — Альт. сценарий 3а1):
Код состояния: 409 Conflict
Тело ответа: JSON с ошибкой. Пример: {"error_code": "SLOT_ALREADY_BOOKED", "message": "Выбранный слот забронирован в системе клиентом"}

#### 3/ Изменение забронированного слота

Набор «вопрос — ответы»:
Вопрос: как фронтенду обновить данные существующего слота (изменить время или мастера), если клиент согласился на перенос?
Ответ: фронтенд отправляет PUT-запрос с ID слота и новыми параметрами. Бэкенд обновляет запись и инициирует отправку СМС.

Определение HTTP-метода:
CRUD операция: Update (обновление)
Безопасный метод: нет
Идемпотентный метод: да, требуется (при повторной отправке того же запроса результат не должен изменяться)
Итоговый метод: PUT

Описание запроса:
Метод: PUT
URL-путь: /api/v1/slots/{slot_id}
Параметры: slot_id
Тело запроса: JSON-объект с новыми данными:
staff_id (Integer, обязательное) - ID нового мастера (или старого, если меняется только время)
start_at (String/DATETIME, обязательное) - новое время начала слота

Набор ответов:
Успешный ответ:
Код состояния: 200 OK
Тело ответа: JSON с обновленным слотом. Пример: {"slot_id": 55, "staff_id": 2, "start_at": "2026-14-25T12:00:00Z", "status": "Free"}

Неуспешный ответ (новое время тоже занято):
Код состояния: 409 Conflict
Тело ответа: JSON с ошибкой. Пример: {"error_code": "SLOT_UNAVAILABLE", "message": "Выбранное новое время также забронировано"}

#### 4/ Отправка СМС о переносе визита

Набор «вопрос — ответы»:
Вопрос: Как бэкенду уведомить клиента об изменении времени или мастера визита?
Ответ: бэкенд формирует текст сообщения с новыми данными визита и отправляет POST-запрос на внешний SMS-шлюз.

Определение HTTP-метода:
CRUD операция: Create (создание задачи на отправку)
Безопасный метод: нет
Идемпотентный метод: нет
Итоговый метод: POST

Описание запроса:
Метод: POST
URL-путь: /api/v1/external/sms/send (эндпоинт внешнего SMS-шлюза)
Параметры: отсутствуют
Тело запроса: JSON-объект:
recipient_phone (String, обязательное) - номер клиента (получен из связки Слот -> Клиент -> Персона)
message_text (String, обязательное) - текст: "Ваш визит перенесен на новое время к мастеру имя нового мастера"

Набор ответов:
Успешный ответ:
Код состояния: 200 OK
Тело ответа: JSON от шлюза. Пример: {"message_id": "msg_998877", "status": "queued"}

Неуспешный ответ:
Код состояния: 400 Bad Request
Тело ответа: JSON с ошибкой шлюза. Пример: {"error_code": "INVALID_PHONE", "message": "Невозможно отправить SMS на данный номер"}
