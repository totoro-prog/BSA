# Exercise 01 — Создание требований и регистрация

## Контекст: выделенный фрагмент задачи

> *«Система должна обеспечивать автоматическую отправку напоминаний клиентам через выбранный клиентом канал связи (Telegram, WhatsApp, VK, СМС) по настроенному менеджером расписанию.»*

**Базовое бизнес-требование:** `BSN-000004` — Автоматизация напоминаний клиентам.

## 2. Функциональные требования к системе

### FUN-000003 — Настройка расписания уведомлений (бэкенд)

| Атрибут                  | Значение                                                     |
| :----------------------- | :----------------------------------------------------------- |
| **ID**                   | FUN-000003                                                   |
| **Краткое наименование** | Настройка расписания уведомлений                             |
| **Описание**             | Система хранит JSON-конфигурацию правил отправки: список интервалов в минутах до визита, шаблон сообщения, каналы по умолчанию |
| **Статус**               | На анализе                                                   |
| **Приоритет**            | Высокий                                                      |
| **Источник**             | USR-000009, USR-000006                                       |
| **Связи**                | FUN-000004, FUN-000005                                       |

**Детализация:**

1. Правила хранятся в таблице `notification_rules` (поля: `id`, `interval_minutes`, `is_active`, `template_text`, `default_channels`).
2. При изменении правил система пересчитывает планируемые отправки для будущих записей.
3. API-эндпоинты:
   - `GET /api/v1/notification-rules` — получить текущие правила
   - `PUT /api/v1/notification-rules` — обновить правила
4. Валидация: интервалы должны быть положительными (≥ 5 минут), минимум одно активное правило.

### FUN-000004 — Автоматическая отправка напоминаний (бэкенд)

| Атрибут                  | Значение                                                     |
| :----------------------- | :----------------------------------------------------------- |
| **ID**                   | FUN-000004                                                   |
| **Краткое наименование** | Автоматическая отправка напоминаний                          |
| **Описание**             | Системный планировщик (cron) каждые 10 минут опрашивает таблицу записей, вычисляет, какие подходят под правила, и отправляет сообщения через адаптеры каналов |
| **Статус**               | На анализе                                                   |
| **Приоритет**            | Высокий                                                      |
| **Источник**             | USR-000007, USR-000008                                       |
| **Связи**                | FUN-000003, FUN-000006                                       |

**Детализация:**

1. Планировщик запускается каждые 10 минут (cron: `*/10 * * * *`).
2. Алгоритм:
   - Выбрать все будущие записи на ближайшие 7 дней (`appointments.start_time > NOW()`)
   - Для каждой записи определить разницу `appointment.start_time - NOW()` в минутах
   - Если разница совпадает с любым активным интервалом из `notification_rules` (с погрешностью ±2 минуты) → инициировать отправку
   - Проверить в таблице `sent_notifications`, не было ли уже отправлено для этой записи + правила (чтобы не дублировать)
3. Отправка:
   - Определить канал клиента из его профиля (`client.channels`)
   - Сформировать сообщение по шаблону, подставив переменные: `{client_name}`, `{master_name}`, `{service}`, `{datetime}`, `{address}`
   - Вызвать соответствующий адаптер (TelegramBotAPI, WhatsAppBusinessAPI, VKAPI, SMSGateway)
4. Обработка ошибок:
   - При ошибке отправки — 3 повтора с шагом 15 минут
   - После 3 неудачных попыток — статус «Ошибка» в журнале
5. Логирование: каждая попытка записывается в таблицу `sent_notifications`.

## 3. Требования к интерфейсам системы (дополнительные к п.2)

### FUN-000007 — API проверки бронирования слота

| Атрибут                  | Значение                                                     |
| :----------------------- | :----------------------------------------------------------- |
| **ID**                   | FUN-000007                                                   |
| **Краткое наименование** | API проверки бронирования                                    |
| **Описание**             | Эндпоинт для проверки, забронирован ли слот клиентом перед созданием (шаг 3 UC02) |
| **Статус**               | На тестировании                                              |
| **Приоритет**            | Высокий                                                      |
| **Источник**             | USR-000001                                                   |
| **Связи**                | FUN-000002                                                   |

**Спецификация:**

- Метод: `GET`

- URL: `/api/v1/slots/check`

- Параметры запроса:

  - `date` (string, обязательный, формат YYYY-MM-DD)
  - `start_time` (string, обязательный, формат HH:MM)
  - `end_time` (string, обязательный, формат HH:MM)
  - `master_id` (string, обязательный)

- Ответ (успех, 200):

  json

  ```
  {
    "is_booked": false,
    "message": "Слот не забронирован клиентом"
  }
  ```

  

- Ответ (забронирован, 200):

  json

  ```
  {
    "is_booked": true,
    "booked_by": "Клиент Иванов И.И.",
    "message": "Слот забронирован клиентом"
  }
  ```

  

- Ответ (ошибка, 400): некорректные параметры

- Ответ (ошибка, 500): внутренняя ошибка сервера

### FUN-000008 — API создания слота

| Атрибут                  | Значение                                                     |
| :----------------------- | :----------------------------------------------------------- |
| **ID**                   | FUN-000008                                                   |
| **Краткое наименование** | API создания слота                                           |
| **Описание**             | Эндпоинт для создания нового слота мастера после успешной проверки (шаг 5 UC02) |
| **Статус**               | В разработке                                                 |
| **Приоритет**            | Высокий                                                      |
| **Источник**             | USR-000001                                                   |
| **Связи**                | FUN-000001                                                   |

**Спецификация:**

- Метод: `POST`

- URL: `/api/v1/slots`

- Тело запроса (JSON):

  json

  ```
  {
    "date": "2026-05-20",
    "start_time": "10:00",
    "end_time": "11:00",
    "master_id": "EMP001"
  }
  ```

  

- Ответ (успех, 201):

  json

  ```
  {
    "slot_id": "SLT001",
    "status": "free",
    "message": "Слот успешно создан"
  }
  ```

  

- Ответ (конфликт, 409): слот уже существует или забронирован

- Ответ (ошибка, 400): невалидные данные (прошедшая дата, некорректное время)

- Ответ (ошибка, 422): мастер не активен или не существует
