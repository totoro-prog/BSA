# Модели и диаграммы

## Кейс 1. Разработка системы онлайн-записи для сети барбершопов

Руководство компании, владеющей сетью барбершопов, запланировало запуск веб-сервиса для дистанционного бронирования визитов. Данный шаг призван привлечь поток новых клиентов, разгрузить персонал и минимизировать ручную работу за счет автоматических оповещений.

Пользоваться платформой смогут как авторизованные пользователи, так и новые посетители без регистрации. В процессе оформления записи клиент выбирает категорию (парикмахерские или косметологические услуги), конкретную процедуру, мастера и подходящее свободное время. Система в автоматическом режиме отправляет напоминания через мессенджеры или СМС (Telegram, WhatsApp, VK) по графику, который настраивает администратор. После визита пользователю предлагается поставить оценку и оставить отзыв.

Со стороны персонала в системе работает менеджер (или несколько сотрудников), который формирует и корректирует расписание мастеров, координирует клиентов в ручном режиме, отмечает выполнение услуг, принимает оплату и передает финансовые данные в бухгалтерию, а также анализирует отзывы и отчетность. Сами мастера имеют доступ к системе исключительно для просмотра своего личного графика и ознакомления с отзывами о своей работе.

## Кейс 2. Создание логистической платформы для службы доставки

В период локдауна резко выросла потребность в оперативной доставке небольших партий товаров из магазинов и заведений общепита. Группа студентов решила организовать собственный логистический стартап. Суть идеи заключается в том, чтобы собирать сведения о заказах, местах сборки и адресах доставки, а затем оперативно распределять их между курьерами через единую онлайн-платформу.

На старте данные от партнеров будут собираться любыми доступными способами и заноситься в единый формат силами оператора, но для курьеров сразу разрабатывается мобильное приложение. Через него исполнитель может изучать список доступных заказов, бронировать их, забирать в точке выдачи и доставлять клиенту, мгновенно меняя статус выполнения в системе.

Для контроля за процессами предусмотрено рабочее место диспетчера, который мониторит курьеров и при необходимости переназначает заказы. Сведения о новых заказах автоматически отправляются во внешнюю бухгалтерскую программу для расчетов с поставщиками. Туда же уходит информация о выполненных доставках для расчета вознаграждения курьеров. Итоговые суммы начислений выгружаются обратно в приложение и отображаются в личном кабинете исполнителя. За регистрацию сотрудников и управление правами доступа отвечает системный администратор.

## Задание 00. Проектирование архитектуры данных: Диаграмма DFD (Уровень 0)

**Инструкция для каждого кейса:**

Постройте диаграмму потоков данных (**Data Flow Diagram**) нулевого уровня, соблюдая следующие требования:

- Четко сформулируйте цель моделирования и укажите контекст схемы (анализ текущих процессов **As-Is** или проектирование целевого состояния **To-Be**).
    
- Графически отобразите ключевые компоненты:
    
    - **Внешние сущности** (прямоугольники) — сторонние системы, роли или контрагенты, взаимодействующие с продуктом.
        
    - **Процессы** (овалы) — операции обработки, преобразования или генерации данных.
        
    - **Потоки данных** (направленные стрелки) — информационные пакеты, передаваемые между элементами.
        
    - **Накопители/Хранилища данных** (параллельные линии) — места системного хранения информации.
        
- **Соблюдайте методологические правила:**
    
    1. Присвойте каждому процессу сквозной порядковый номер (строго целые числа: 1, 2, 3...).
        
    2. Формируйте названия процессов по классической схеме: `[Глагол в инфинитиве] + [Объект]`(например, _«Проверить баланс»_).
        
    3. Используйте понятные бизнесу термины для объектов, избегая сугубо технических названий таблиц БД.
        
    4. Исключите прямые связи между процессами — они должны обмениваться информацией только через накопители данных.
        
    5. Потоки данных между хранилищами и внешними сущностями всегда должны проходить через процесс-посредник.
        
    6. Помните, что DFD отражает логику движения данных, а не временную последовательность шагов. Схема должна оставаться легко читаемой.
        
- Результат сохраните в файл **ex00_[префикс_продукта]_dfd.xxx** (расширение зависит от выбранного инструмента проектирования).
    

## Задание 01. Моделирование бизнес-процессов: Кросс-функциональная схема (Swimlane Diagram)

**Инструкция для каждого кейса:**

Спроектируйте диаграмму «плавательные дорожки» для визуализации логики выполнения процессов:

- Укажите назначение схемы и анализируемый контур процессов (**As-Is** или **To-Be**).
    
- Определите ключевых участников процесса: роли стейкхолдеров, внутренние подразделения компании или интегрируемые ИТ-системы.
    
- Выделите для каждого участника персональную функциональную дорожку (горизонтальный или вертикальный трек).
    
- Разложите процесс на последовательные шаги (прямоугольники) и распределите их по дорожкам соответствующих исполнителей.
    
- Соедините шаги стрелками перехода, отображая логическую последовательность действий.
    
- Для ветвления логики используйте логические развилки (ромбы), обязательно подписывая условия на выходящих из них стрелках.
    
- Обозначьте точки входа и выхода из процесса с помощью стартовых и финальных терминаторов (кругов).
    
- Готовую работу сохраните под именем **ex01_[префикс_продукта]_swd.xxx**.
    

## Задание 02. Расширение кросс-функциональной модели: Интеграция системных артефактов

**Инструкция для каждого кейса:**

Дополните созданную ранее Swimlane-диаграмму объектами и сущностями, которые создаются или изменяются в ходе выполнения процесса:

- Сделайте копию схемы из Задания 01 и присвойте ей новое имя.
    
- Зафиксируйте цель построения и область анализа (**As-Is / To-Be**).
    
- Добавьте на дорожки исполнителей генерируемые ими артефакты: отчетные документы, программный код, зарегистрированные баг-репорты, входящие запросы или уведомления.
    
- Соедините каждый артефакт с шагом процесса, на котором он создается или редактируется, используя пунктирные линии связи.
    
- При необходимости группируйте артефакты в единые логические пулы (контейнеры), если они создаются одной и той же ролью в рамках одной технологической операции.
    
- Итоговый файл назовите **ex02_[префикс_продукта]_adswd.xxx**.
    

## Задание 03. Моделирование жизненного цикла сущностей: Диаграмма состояний (State Diagram)

**Инструкция для каждого кейса:**

Спроектируйте конечно-автоматную модель для системного объекта:

- Выберите ключевую бизнес-сущность предметной области, обладающую сложным жизненным циклом. _Ограничение для Кейса 1: выберите любой объект, кроме сущности «Заявка на услугу»_.
    
- Пропишите цель создания модели и её контекст.
    
- Постройте диаграмму состояний выбранного объекта (State Machine Diagram):
    
    - Сформулируйте и отобразите на схеме все валидные статусы (состояния) объекта.
        
    - Соедините статусы направленными стрелками переходов.
        
    - Возле каждой стрелки четко укажите триггер (событие или действие), который переводит объект на следующий этап, и зафиксируйте инициатора этого события.
        
    - При наличии альтернативных путей перехода пропишите контролирующие условия (guard conditions) в квадратных скобках.
        
    - Обязательно укажите маркеры начала и завершения жизненного цикла сущности.
        
- Экспортируйте схему в файл **ex03_[префикс_продукта]_dst.xxx**.
    

## Задание 04. Документирование жизненного цикла: Матрица переходов (State Table)

**Инструкция для каждого кейса:**

Сформируйте аналитическую таблицу состояний для объектов системы:

- Выделите сущности проекта, проходящие через цепочку изменений.
    
- Для тех объектов, которые не были детально визуализированы в форме графической диаграммы в Задании 03, составьте подробную текстовую матрицу переходов.
    
- Укажите название анализируемого объекта, цель формирования таблицы и её рамки (**As-Is / To-Be**).
    
- Структурируйте таблицу, включив в строки следующие параметры:
    
    - Исходное (предыдущее) состояние сущности.
        
    - Инициирующее событие/операция и роль исполнителя, совершающего действие.

    - Целевое (последующее) состояние, в которое переходит объект.
        
    - Явные отметки о точках инициализации (создания) и терминации (архивации/удаления) объекта в системе.
        
- Отчет сохраните под именем **ex04_[префикс_продукта]_tst.xxx**.
